## 2. Redundancy Analysis

### 2.1 Potential Overlap: Line vs Resonator (MINOR CONCERN)

**Observation:** Both morphemes claim to encode "transformation":
- Line: "Encodes: Flow, **transformation**, direction"
- Resonator: "Encodes: **Transformation**, decision, routing"

**Analysis:** Despite the shared term, these are semantically distinct:

| Aspect | Line | Resonator |
|--------|------|-----------|
| Graph role | Edge | Node |
| Transformation type | In-transit (mapping during transfer) | At-rest (processing at a point) |
| Analogy | Pipe (carries, may filter) | Filter (transforms, may route) |
| Primary purpose | Connection | Processing |

**Verdict:** NOT REDUNDANT. The overlap in terminology is misleading. Recommend clarifying spec language:
- Line: "Encodes: Flow, **transfer**, direction" (change "transformation" to "transfer")
- This preserves Line as the edge/connection concept and Resonator as the vertex/processing concept

### 2.2 Potential Overlap: Bloom vs Grid (NO CONCERN)

**Observation:** Both relate to containment/structure.

**Analysis:** These are clearly distinct:
- **Bloom**: Spatial containment, scope boundary, interface definition (synchronous)
- **Grid**: Knowledge persistence, memory structure, state storage (temporal)

A Bloom contains morphemes spatially; a Grid stores knowledge temporally. A Bloom can contain a Grid.

**Verdict:** NOT REDUNDANT. Clearly distinct concepts.

### 2.3 Potential Overlap: Seed vs Resonator decision points (NO CONCERN)

**Observation:** Seed has `seedType: "decision"` but Resonator encodes "decision".

**Analysis:** 
- Seed decision: A decision datum (the result of a decision, or a decision to be made)
- Resonator decision: The act of deciding (a transformation that routes)

**Verdict:** NOT REDUNDANT. Seed is noun (decision point), Resonator is verb (deciding).

### 3.2 Potentially Missing: Event/Signal Morpheme (CONSIDER)

**Gap Identified:** Asynchronous, event-driven patterns are not first-class.

**Current Workaround:** Events are modeled as:
- Lines with conditions (triggered connections)
- Resonators in reception mode (∇ orientation)

**Assessment:** For the target domain (agentic workflows, knowledge graphs), the current approach is adequate because:
1. Most workflows are request-response or polling-based
2. Events can be modeled as Seeds with temporal metadata flowing through Lines
3. Adding a 7th morpheme increases complexity for marginal benefit

**Recommendation:** DEFER. Document the pattern for modeling events using existing morphemes rather than adding an Event morpheme. If real implementations consistently struggle to model event-driven systems, reconsider.

### 3.3 Potentially Missing: Constraint/Rule Morpheme (REJECT)

**Gap Identified:** No explicit morpheme for invariants or business rules.

**Assessment:** This is correctly handled at the grammar level (G1-G5), not the morpheme level. Morphemes are structural primitives; constraints are relationships between them.

**Recommendation:** REJECT. Constraints belong in grammar rules, not morphemes.

### 3.4 Potentially Missing: Alias/Reference Morpheme (REJECT)

**Gap Identified:** No way to create symbolic references or aliases to other morphemes.

**Assessment:** This is an implementation concern (handled via IDs) not a semantic encoding concern. Adding a reference morpheme would create indirection that complicates the "state is structural" principle.

**Recommendation:** REJECT. IDs handle this at implementation level.

## 4. Justification for the Six Morphemes

The six morphemes map to fundamental dimensions of system description:

```
           SPACE                    TIME
             │                        │
    ┌────────┼────────┐      ┌────────┼────────┐
    │        │        │      │        │        │
  Seed    Bloom     Grid   Helix    Line   Resonator
    │        │        │      │        │        │
 (point) (boundary) (memory) (iteration) (flow) (transform)
    │        │        │      │        │        │
    └────────┴────────┘      └────────┴────────┘
         STATIC                  DYNAMIC
```

### Dimensional Coverage:

1. **Seed** - The atomic unit (WHERE something exists)
2. **Line** - The connection (HOW things relate)  
3. **Bloom** - The boundary (WHAT scope contains)
4. **Resonator** - The process (WHAT transformation occurs)
5. **Grid** - The memory (WHAT persists)
6. **Helix** - The iteration (HOW time progresses)

### Why Not Fewer?

- **Merge Seed + Resonator?** No - conflates data with processing
- **Merge Line + Helix?** No - conflates connection with iteration
- **Merge Bloom + Grid?** No - conflates spatial containment with temporal persistence
- **Remove any one?** Each encodes a distinct dimension that cannot be expressed by combinations of others

### Why Not More?

Each proposed additional morpheme (Event, Constraint, Reference) can be expressed as:
- A pattern using existing morphemes, OR
- A property of existing morphemes, OR
- A grammar rule

Adding morphemes increases combinatorial complexity (grammar rules scale with morpheme count) without proportional expressive power.

/**
 * Codex Signum — Morpheme Type Definitions
 *
 * The six fundamental forms from which all expressions compose.
 * These are immutable. Their meanings are fixed across all versions,
 * all implementations, all scales.
 *
 * Morpheme Selection Rationale:
 * The six morphemes cover orthogonal dimensions of system description:
 * - Seed: WHERE (point of existence)
 * - Line: HOW CONNECTED (relationship/transfer)
 * - Bloom: WHAT SCOPE (boundary/containment)
 * - Resonator: WHAT PROCESS (transformation/routing)
 * - Grid: WHAT PERSISTS (memory/structure)
 * - Helix: HOW TIME FLOWS (iteration/learning)
 *
 * This set is minimal (no morpheme can be removed without losing
 * expressiveness) and complete (no additional morpheme is required
 * to model agentic workflows, knowledge graphs, or distributed systems).
 *
 * @see codex-signum-v3.0.md §The Six Morphemes
 * @module codex-signum-core/types/morphemes
 */

// ============ MORPHEME KINDS ============

/**
 * The six morpheme types — dimensional channels of the encoding space.
 * Each defines a distinct encoding dimension:
 * - Seed: lifecycle stage (the atomic datum)
 * - Line: connectivity (the transfer channel)
 * - Bloom: scope (the containment boundary)
 * - Resonator: transformation (the processing node)
 * - Grid: structure (the persistent memory)
 * - Helix: temporality (the iteration cycle)
 */
export type MorphemeKind =
  | "seed"
  | "line"
  | "bloom"
  | "resonator"
  | "grid"
  | "helix";

// ============ INTEGRATION STATE ============

/**
 * Integration state lifecycle for any morpheme.
 * Created → Dormant → Connected → Active → [Degraded → Recovering | Archived]
 */
export type IntegrationState =
  | "created"
  | "dormant"
  | "connected"
  | "active"
  | "degraded"
  | "recovering"
  | "archived";

// ============ BASE MORPHEME ============

/**
 * Base properties shared by all morphemes.
 * Every morpheme carries the three state dimensions.
 */
export interface MorphemeBase {
  /** Unique identifier */
  id: string;
  /** Morpheme kind — determines structural role */
  kind: MorphemeKind;
  /** Human-readable name */
  name: string;
  /** Integration lifecycle state */
  state: IntegrationState;
  /** Creation timestamp */
  createdAt: Date;
  /** Last updated timestamp */
  updatedAt: Date;
  /** Optional description of purpose */
  description?: string;
  /** Optional tags for classification */
  tags?: string[];
}

// ============ SEED (•) ============

/**
 * Seed — Point/Singularity
 * Encodes: Origin, instance, datum, coherent unit.
 * The atomic unit — a piece of data, a function instance, a decision point.
 *
 * Seed is the NOUN — what exists at a point in the system.
 * Contrast with Resonator which is the VERB — what transforms at a point.
 *
 * A Seed with no inbound or outbound Lines is Dormant.
 */
export interface Seed extends MorphemeBase {
  kind: "seed";
  /** Role this seed plays in a composition */
  role: string;
  /** The type of datum or function this seed represents */
  seedType: "function" | "datum" | "decision" | "input" | "output";
  /** Whether this seed accepts input */
  accepts?: string;
  /** Whether this seed produces output */
  produces?: string;
}

// ============ LINE (→) ============

/**
 * Line — Vector/Connection
 * Encodes: Flow, transfer, direction.
 *
 * Line is the EDGE in a graph — it connects morphemes and carries data.
 * Data may be filtered or mapped during transfer, but the primary role
 * is connection, not processing.
 *
 * Contrast with Resonator which is the NODE that performs transformation.
 * Line = pipe (carries, may filter)
 * Resonator = filter (transforms, may route)
 *
 * Connection requires intent (G1).
 */
export type LineDirection = "forward" | "return" | "parallel" | "bidirectional";

export interface Line extends MorphemeBase {
  kind: "line";
  /** Source morpheme ID */
  from: string;
  /** Target morpheme ID */
  to: string;
  /** Direction encodes relationship (G2) */
  direction: LineDirection;
  /** Optional condition for when this line activates */
  condition?: string;
  /** Connection weight — relationship strength */
  weight: number;
}

// ============ BLOOM (○) ============

/**
 * Bloom — Circle/Boundary
 * Encodes: Scope, boundary, context.
 *
 * Bloom is SPATIAL containment — what exists within a boundary at any moment.
 * It defines interfaces (what can cross the boundary) and scope (what's inside).
 *
 * Contrast with Grid which is TEMPORAL containment — what persists over time.
 * Bloom = module boundary (synchronous scope)
 * Grid = database (persistent memory)
 *
 * Containment creates scope (G3).
 */
export type BloomShape = "open" | "closed";

export interface Bloom extends MorphemeBase {
  kind: "bloom";
  /** Open C-shape = receptive interface; Closed = protected scope */
  shape: BloomShape;
  /** IDs of morphemes contained within this bloom */
  contains: string[];
  /** Interface definition — where external connection happens */
  interface: {
    inputs: Array<{ id: string; type: string; accepts: string }>;
    outputs: Array<{ id: string; type: string; produces: string }>;
    exposes: Array<{
      id: string;
      type: string;
      access: "read" | "write" | "observe";
    }>;
  };
  /** Dependencies this bloom requires */
  dependencies: {
    requiresModels: string[];
    requiresGrids: string[];
    requiresExternal: string[];
  };
}

// ============ RESONATOR (Δ) ============

/**
 * Resonator — Triangle/Process
 * Encodes: Transformation, decision, routing.
 *
 * Resonator is the PROCESSING NODE — where data is transformed.
 * It is the VERB of the system.
 *
 * Contrast with Line which is the EDGE that transfers data.
 * Contrast with Seed which is the NOUN that exists as data.
 *
 * Orientation matters: up (Δ) = emission/output, down (∇) = reception/input.
 *
 * Event-driven patterns: Model events as Seeds flowing through Lines into
 * reception-oriented Resonators (∇). The Resonator handles the event
 * processing; the Line handles the event delivery.
 */
export type ResonatorOrientation = "up" | "down";

export interface Resonator extends MorphemeBase {
  kind: "resonator";
  /** Apex up (Δ) = emission/output; Apex down (∇) = reception/input */
  orientation: ResonatorOrientation;
  /** The transformation role this resonator performs */
  role: string;
  /** Optional mechanism description */
  mechanism?: string;
}

// ============ GRID (□) ============

/**
 * Grid — Square/Structure
 * Encodes: Network, schema, knowledge structure.
 * Where persistent memory enables persona emergence.
 *
 * Grid is TEMPORAL containment — what persists across time.
 *
 * Contrast with Bloom which is SPATIAL containment — what exists within
 * a boundary at any moment.
 * Grid = database (persistent memory)
 * Bloom = module boundary (synchronous scope)
 */
export type GridType = "persistent" | "ephemeral" | "shared";

export interface Grid extends MorphemeBase {
  kind: "grid";
  /** Persistence type */
  gridType: GridType;
  /** Schema reference for the knowledge structure */
  schemaRef?: string;
  /** Memory stratum this grid operates at (1-4) */
  stratum?: 1 | 2 | 3 | 4;
}

// ============ HELIX (🌀) ============

/**
 * Helix — Spiral/Evolution
 * Encodes: Recursion, iteration, temporal flow, learning.
 *
 * Helix captures HOW TIME PROGRESSES in the system — whether through
 * tight correction loops, learning cycles, or evolutionary adaptation.
 *
 * Mode is INFERRED from temporal constant and containment context,
 * not declared. Convention:
 * - Correction: within single execution (seconds to minutes)
 * - Learning: across executions (hours to weeks)
 * - Evolutionary: across ecosystem (weeks to months)
 */
export type HelixMode = "correction" | "learning" | "evolutionary";

export interface Helix extends MorphemeBase {
  kind: "helix";
  /** Inferred mode based on temporal constant and context */
  mode: HelixMode;
  /** ID of the morpheme this helix feeds into */
  feedsInto: string;
  /** Optional source grid for learning data */
  source?: string;
  /** Maximum iterations for correction helixes */
  maxIterations?: number;
}

// ============ UNION TYPE ============

/**
 * Any morpheme — discriminated union on `kind`.
 */
export type Morpheme = Seed | Line | Bloom | Resonator | Grid | Helix;

// ============ GRAMMAR COMPLIANCE ============

/**
 * Grammar rule compliance for a composition.
 * G1-G5 are the five grammar rules.
 */
export interface GrammarCompliance {
  /** G1: Proximity — Connection requires intent */
  G1_proximity: boolean;
  /** G2: Orientation — Direction encodes flow */
  G2_orientation: boolean;
  /** G3: Containment — Enclosure creates scope */
  G3_containment: boolean;
  /** G4: Flow — Light movement is data transfer */
  G4_flow: boolean;
  /** G5: Resonance — Alignment enables composition */
  G5_resonance: boolean;
  /** List of specific violations found */
  violations: string[];
}

// ============ MORPHEME COMPLETENESS DOCUMENTATION ============

 * The morphemes were selected to cover orthogonal dimensions of system description:

 * 1. SEED (•) - Point existence (WHERE)
 *    Cannot be merged with Resonator: Seed is datum, Resonator is process
 *
 * 2. LINE (→) - Connection/transfer (HOW CONNECTED)
 *    Cannot be merged with Helix: Line is spatial connection, Helix is temporal iteration
 *
 * 3. BLOOM (○) - Spatial containment (WHAT SCOPE)
 *    Cannot be merged with Grid: Bloom is boundary, Grid is memory
 *
 * 4. RESONATOR (Δ) - Transformation (WHAT PROCESS)
 *    Cannot be merged with Seed: Resonator is verb, Seed is noun
 *
 * 5. GRID (□) - Temporal persistence (WHAT PERSISTS)
 *    Cannot be merged with Bloom: Grid is across time, Bloom is at a time
 *
 * 6. HELIX (🌀) - Iteration/learning (HOW TIME FLOWS)
 *    Cannot be merged with Line: Helix is temporal, Line is spatial
 *
 * Patterns NOT given dedicated morphemes:
 * - Events: Model as Seed + Line + Resonator(∇)
 * - Constraints: Express via Grammar Rules G1-G5
 * - References: Handled by ID system at implementation level
 */
export const MORPHEME_RATIONALE = {

  minimal: true, // No morpheme can be removed without losing expressiveness
