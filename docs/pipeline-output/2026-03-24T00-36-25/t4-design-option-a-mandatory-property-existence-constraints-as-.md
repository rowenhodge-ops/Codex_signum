# Design Option A — Mandatory Property Existence Constraints as Structural Gatekeeping

**Document:** R-63 Design Option A Analysis
**Task:** t4
**Status:** Design analysis — Architect review required
**References:** cs-v5.0.md (Constitutional Coupling, A6 Minimal Authority), t1 (Threat Model), t2 (Enforcement Audit), t3 (Capabilities Survey)

---

## 1. Core Concept

Every morpheme-labeled node in the graph is required by Neo4j property existence constraints and node key constraints to carry a set of **governance provenance properties** at the moment of creation. If a raw Cypher `CREATE` or `MERGE` omits any of these properties, the transaction fails at the database engine level — before commit, without application layer involvement.

The structural thesis: if the governance layer is the *only* system that knows how to produce valid values for these properties, then the database constraint becomes a **structural proxy** for governance participation. The attacker can write the Cypher, but cannot satisfy the constraint without going through the governed path.

The central question this document must answer honestly: **is this thesis sound, or does it collapse under the fake-value attack?**

## 2. Governance Property Schema Design

### 2.1 Mandatory Governance Properties

Every morpheme node (Seed, Bloom, Resonator, Grid, Helix) must carry these properties:

| Property | Type (semantic) | Purpose | Source of Truth |
|---|---|---|---|
| `_gov_instantiation_id` | UUID v4 string | Links to a specific Instantiation protocol record. Must correspond to an actual `InstantiationRecord` node. | Generated by `instantiateMorpheme()` at creation time |
| `_gov_provenance_epoch` | ISO-8601 datetime string | Timestamp of governance-layer creation. Establishes temporal ordering in the provenance chain. | Set by `instantiateMorpheme()` from server clock |
| `_gov_contains_path` | String (dot-delimited path) | The containment path from root to this node's parent at creation time. Encodes the G3 placement decision. Example: `constitutional-bloom.M-9.M-9.7` | Computed by `instantiateMorpheme()` from parent traversal |
| `_gov_schema_version` | Integer | Schema version under which this node was created. Enables migration path analysis. | Hardcoded constant in Instantiation Resonator |
| `_gov_write_token` | String | Cryptographic token binding the creation to a specific governance transaction. See §4 for analysis. | Generated by governance layer per-transaction |

The `_gov_` prefix is a namespace convention: it signals that these properties are governance infrastructure, not domain data. Application queries should never filter on them. Governance audits should always verify them.

### 2.2 Per-Label Additional Requirements

Beyond the five universal governance properties, each morpheme label retains its existing domain-required properties (from R-39):

| Label | Existing Required Properties | New Governance Properties |
|---|---|---|
| **Seed** | `id`, `content`, `seedType`, `status` | `_gov_instantiation_id`, `_gov_provenance_epoch`, `_gov_contains_path`, `_gov_schema_version`, `_gov_write_token` |
| **Bloom** | `id`, `type`, `status` | Same five governance properties |
| **Resonator** | `id` (currently only uniqueness) | Same five + new existence constraints for `name`, `content`, `type`, `status` |
| **Grid** | `id` (currently only uniqueness) | Same five + new existence constraints for `name`, `content`, `type`, `status` |
| **Helix** | `id` (currently only uniqueness) | Same five + new existence constraints for `name`, `content`, `mode`, `status` |

**Design note:** The t2 audit revealed that Resonator, Grid, and Helix currently have only uniqueness constraints on `id` — no property existence constraints at all. This means a raw Cypher attacker can create these node types with *nothing but an id*. The governance property scheme must close this gap simultaneously with the governance properties themselves.

## 4. Fake-Value Attack Analysis

This is the critical section. The governance property existence constraints guarantee that an attacker *cannot create a morpheme node without providing all governance properties*. But they do not verify the *validity* of the values provided. This section systematically examines whether an attacker can provide arbitrary but constraint-satisfying values.

### 4.1 Attack: Arbitrary `_gov_instantiation_id`

**Attack Cypher:**
```cypher
CREATE (s:Seed {
  id: 'injected-seed-42',
  name: 'Looks legitimate',
  content: 'Injected payload',
  seedType: 'observation',
  status: 'active',
  _gov_instantiation_id: 'fake-uuid-' + toString(randomUUID()),
  _gov_provenance_epoch: datetime(),
  _gov_contains_path: 'constitutional-bloom.M-9',
  _gov_schema_version: 1,
  _gov_write_token: 'fake-token-' + toString(randomUUID())
})
```

**Result: The node is created successfully.** The existence constraints are satisfied. The uniqueness constraints are satisfied (fresh UUIDs). The node has no CONTAINS or INSTANTIATES wiring, but it has all required properties.

**Assessment: The existence constraint alone does NOT prevent this attack.**

The attacker's `_gov_instantiation_id` does not correspond to any `InstantiationRecord` node. But Neo4j has no foreign key constraint mechanism. The database cannot verify that the UUID references an existing record.

### 4.2 Attack: Replayed `_gov_instantiation_id` from Legitimate Node

**Attack Cypher:**
```cypher
// Step 1: Read a legitimate node's governance properties
MATCH (s:Seed) WHERE s.status = 'active' RETURN s._gov_instantiation_id LIMIT 1
// → returns 'inst:abc123'

// Step 2: Attempt to create with copied ID
CREATE (s:Seed {
  id: 'injected-seed-replay',
  _gov_instantiation_id: 'inst:abc123',
  ...
})
```

**Result: Transaction fails.** The `_gov_instantiation_id` uniqueness constraint rejects the duplicate. This is the one structural defence that works here — each instantiation ID can only be used once per label.

**Assessment: Uniqueness constraint prevents ID-replay attacks.** But the attacker simply generates a fresh UUID that has never been used. Uniqueness proves the ID hasn't been *duplicated*, not that it's *legitimate*.

### 4.3 Attack: Fake `_gov_contains_path`

The `_gov_contains_path` property records which containment path the governance layer placed the node under. An attacker can provide any plausible-looking path string: `"constitutional-bloom.M-9.7"`. The database cannot verify this path corresponds to an actual CONTAINS chain in the graph.

**Assessment: Entirely bypassable.** The path is a denormalized string. It has no structural relationship to the actual graph topology.

### 4.4 Attack: Fake `_gov_write_token`

If the write token is simply a random string, the attacker generates their own random string. The uniqueness constraint ensures tokens aren't replayed, but freshly-minted fake tokens satisfy the constraint.

**Assessment: A plain random string offers zero defence.** The write token only becomes meaningful if it is *cryptographically unforgeable* — i.e., if the database can verify its authenticity without calling the application layer. See §5.2 for the structural implications.

### 4.5 Attack: Empty-String Governance Properties

```cypher
CREATE (s:Seed {
  id: 'injected-empty-gov',
  content: 'payload',
  seedType: 'task',
  status: 'active',
  _gov_instantiation_id: '',
  _gov_provenance_epoch: '',
  _gov_contains_path: '',
  _gov_schema_version: 0,
  _gov_write_token: ''
})
```

**Result: Succeeds.** `IS NOT NULL` does not reject empty strings. The node is created with governance properties that have zero semantic content.

**Assessment: Existence constraints do not enforce semantic validity.** This is the same weakness identified in t2 for `content`: the application checks `trim() === ""` but the database does not.

### 4.6 Fake-Value Attack Verdict

**The fake-value attack succeeds against pure existence constraints.** An attacker with Class A access (raw Cypher, non-admin) can:

1. Provide any non-null value for every governance property
2. Generate unique UUIDs that satisfy uniqueness constraints
3. Construct plausible-looking but entirely fabricated governance metadata
4. Create nodes that are structurally indistinguishable from legitimate nodes by property inspection alone

**Existence constraints raise the bar** (the attacker must know the property names and provide values for all of them) but **do not create a structural invariant** that distinguishes governed from ungoverned writes.

## 5. Referential Integrity Gap

### 5.1 The Core Problem: Neo4j Has No Foreign Key Constraints

In a relational database, a foreign key constraint on `_gov_instantiation_id` would require that the value exists as a primary key in an `InstantiationRecord` table. The database engine itself would reject any write where the referenced record does not exist.

**Neo4j has no equivalent mechanism.** There is no DDL construct of the form:

```
-- THIS DOES NOT EXIST IN NEO4J
CREATE CONSTRAINT seed_gov_fk IF NOT EXISTS
  FOR (s:Seed) REQUIRE EXISTS ((:InstantiationRecord {id: s._gov_instantiation_id}))
```

This is the fundamental referential integrity gap. The governance properties can *name* a related record, but the database cannot *enforce* that the record exists.

### 5.2 Mitigation Options for the Referential Integrity Gap

#### Option 5.2.A — APOC Trigger as Foreign Key Proxy

An APOC trigger (`apoc.trigger.add`) fires before or after commit and can execute arbitrary Cypher, including existence checks:

```
// Conceptual — not implementation code
ON CREATE of any :Seed node →
  MATCH (ir:InstantiationRecord {id: newNode._gov_instantiation_id})
  WHERE ir IS NULL → ROLLBACK
```

**Strengths:**
- Pre-commit enforcement: the transaction rolls back if the InstantiationRecord doesn't exist
- Fires on every write, including raw Cypher
- The attacker would need to create a plausible InstantiationRecord first (raising the bar significantly)

**Weaknesses:**
- APOC is a plugin, not core Neo4j. Not available on all deployment targets (notably AuraDB Free).
- A Class B adversary (admin) can disable the trigger.
- The trigger itself must be secured — if the attacker can call `apoc.trigger.remove`, the defence evaporates.
- The trigger Cypher executes in the same transaction context, adding latency to every morpheme write.
- The trigger is procedural logic, which is conceptually close to the "monitoring overlay" the spec forbids — though it fires synchronously pre-commit, which is structurally distinct from async scanning.

**Classification:** Structural enforcement (pre-commit, synchronous, transaction-blocking) — but **plugin-dependent** and **admin-removable**. It is NOT a monitoring overlay because it blocks the write, but it IS an enforcement layer that depends on a component outside the database engine proper.

#### Option 5.2.B — InstantiationRecord as Node Key with Linked Write Pattern

Instead of a foreign key, require the governance write pattern to create the `InstantiationRecord` and the morpheme node **in the same transaction**, linked by a mandatory relationship:

```
// The governed write pattern (conceptual):
CREATE (ir:InstantiationRecord {id: $govId, ...})
CREATE (s:Seed {id: $nodeId, _gov_instantiation_id: $govId, ...})
CREATE (s)-[:GOVERNED_BY]->(ir)
```

The `InstantiationRecord` itself has constraints:
```cypher
CREATE CONSTRAINT ir_id_unique IF NOT EXISTS
  FOR (ir:InstantiationRecord) REQUIRE ir.id IS UNIQUE

CREATE CONSTRAINT ir_hmac_required IF NOT EXISTS
  FOR (ir:InstantiationRecord) REQUIRE ir._hmac IS NOT NULL

CREATE CONSTRAINT ir_hmac_unique IF NOT EXISTS
  FOR (ir:InstantiationRecord) REQUIRE ir._hmac IS UNIQUE
```

**But this does not solve the problem.** The attacker can create their own `InstantiationRecord` node with fake values, then reference it. We've just moved the perimeter from "who can create Seed nodes" to "who can create InstantiationRecord nodes" — the same problem at a different label.

**Classification:** Perimeter displacement, not perimeter enforcement. The referential integrity gap recurses.

#### Option 5.2.C — Cryptographic Write Token (HMAC-Based)

The `_gov_write_token` property contains an HMAC computed over the node's governance properties using a secret key held only by the application layer:

```
token = HMAC-SHA256(
  key = APPLICATION_SECRET,
  message = _gov_instantiation_id || _gov_provenance_epoch || _gov_contains_path || node.id
)
```

**The honest analysis:**

This token *cannot be verified by Neo4j natively*. Neo4j has no built-in HMAC verification function. The token becomes verifiable only when:

1. An APOC trigger calls a custom procedure that verifies the HMAC (requires APOC + custom procedure)
2. A query-time check in the application layer verifies HMACs on read (monitoring-pattern — violates requirement (b))
3. A Neo4j Transaction Event Handler (Java plugin) verifies the HMAC pre-commit (requires Enterprise + custom plugin deployment)

**Without verification at write time, the HMAC is security theater.** The attacker provides any 64-character hex string. It satisfies the NOT NULL and UNIQUE constraints. Nobody checks it until the application reads it later — which is detection, not enforcement.

**With APOC-based verification at write time:** The trigger computes the expected HMAC from the other governance properties and the secret, compares it to the provided token. If they don't match, rollback. This is the strongest mitigation available, but it depends entirely on APOC availability and the secrecy of the HMAC key.

**If the application layer is compromised** (requirement (c)), the HMAC key is compromised. The attacker can compute valid HMACs for any node they create. The token becomes worthless.

**Classification:** Defence-in-depth against Class A adversary with non-compromised application. Useless against Class A adversary with compromised application (which is explicitly in scope per requirement (c)).

#### Option 5.2.D — Append-Only Provenance Chain with Sequential Constraint

Each `InstantiationRecord` carries a monotonically increasing sequence number and the hash of the previous record:

```
InstantiationRecord {
  id: 'ir:00042',
  _sequence: 42,
  _prev_hash: 'sha256-of-ir:00041',
  _self_hash: 'sha256-of-this-record',
  ...
}
```

A uniqueness constraint on `_sequence` prevents sequence gaps from being filled, and the hash chain means inserting a record retroactively would require recomputing all subsequent hashes.

**Assessment:** This creates a tamper-evident structure (forgery is detectable after the fact) but not a tamper-proof structure (the database cannot verify hash chain integrity at write time without triggers). A forged record at the head of the chain is indistinguishable from a legitimate record until someone walks the chain and checks hashes — which is detection, not enforcement.

**Classification:** Tamper-evidence, not tamper-prevention. Useful for forensics. Does not satisfy requirement (a).

### 5.3 Referential Integrity Gap Summary

| Mitigation | Prevents Fake Values? | Survives App Compromise? | Requires Plugin? | Monitoring Overlay? |
|---|---|---|---|---|
| 5.2.A APOC Trigger FK | Yes (if trigger active) | Yes (trigger is DB-layer) | Yes (APOC) | No (pre-commit) |
| 5.2.B Linked Node Pattern | No (perimeter displacement) | N/A | No | No |
| 5.2.C HMAC Write Token | Only with trigger verification | No (key compromise) | Yes (APOC or Java plugin) | No (if pre-commit) |
| 5.2.D Hash Chain | No (detection only) | Yes (chain is structural) | No | Yes (requires walking chain) |

**Honest conclusion:** No combination of property existence constraints solves the referential integrity gap without a trigger mechanism or custom plugin. The gap is fundamental to Neo4j's architecture.

## 6. Performance Impact Analysis

### 6.1 Constraint Evaluation Overhead

**Property existence constraints** are evaluated during the transaction commit phase. Neo4j checks that each constrained property is present and non-null. This is a memory-local operation — no index lookup, no disk I/O.

**Per-node overhead:** ~0.01–0.05ms per existence constraint check. With 10 existence constraints per morpheme node (5 governance + ~5 domain), the total per-node overhead is approximately **0.1–0.5ms**.

**Uniqueness constraints** require an index lookup to verify no duplicate exists. For `_gov_instantiation_id` and `_gov_write_token`, each requires one B-tree index probe.

**Per-node overhead for uniqueness:** ~0.1–0.5ms per uniqueness check (index probe), totaling ~0.2–1.0ms for two governance uniqueness constraints.

**Combined overhead per morpheme creation:** ~0.3–1.5ms additional latency. For the current Codex Signum workload profile (estimated <100 morpheme creations per pipeline run), this is negligible.

### 6.2 Index Storage Overhead

Each uniqueness constraint creates a backing index. Two new uniqueness-constrained properties across 5 labels = 10 new indexes. Each index stores one entry per node of that label.

**Estimated storage:** For a graph with 10,000 morpheme nodes, 10 additional indexes storing UUID-sized values ≈ 10 × 10,000 × 64 bytes ≈ 6.4 MB. Negligible.

### 6.3 Write Amplification on MERGE

The existing `instantiateMorpheme()` uses `MERGE` with `ON CREATE SET`. Adding 5 governance properties to the `ON CREATE SET` clause adds 5 property writes per node creation. Since these properties are set only at creation time (not on every MERGE match), the write amplification is limited to the creation path.

**Net assessment: Performance impact is negligible** for the expected workload. The overhead becomes relevant only at >10,000 node creations per second sustained, which is far beyond the Codex Signum operating envelope.

### 6.4 AuraDB Free Tier Compatibility

The t2 audit notes that AuraDB Free tier may reject some constraint types. Property existence constraints (`IS NOT NULL`) are supported on AuraDB Free. Uniqueness constraints are supported on AuraDB Free. Node key constraints are supported on AuraDB Free.

**Assessment: All proposed constraints are compatible with AuraDB Free tier.** The existing `migrateSchema()` error-handling pattern (log and continue if tier rejects) provides graceful degradation, but for governance constraints, degradation should be FATAL — if the constraints cannot be applied, the perimeter is not established. The migration function should distinguish between governance-critical and convenience constraints.

## 7. Compatibility with Existing Instantiation Protocol

### 7.1 Required Changes to `instantiateMorpheme()`

The function must be modified to:

1. **Generate `_gov_instantiation_id`:** A UUID v4 generated at the start of the function, before any database call. This becomes the immutable identifier for this governance transaction.

2. **Compute `_gov_contains_path`:** After the parent existence check (Step 2), traverse the CONTAINS chain upward to build the containment path string. This is a read query:
   ```
   MATCH path = (root)-[:CONTAINS*]->(parent {id: $parentId})
   WHERE NOT ()-[:CONTAINS]->(root)
   RETURN [n IN nodes(path) | n.id] AS pathIds
   ```
   The path is then joined: `pathIds.join('.')`.

3. **Set `_gov_schema_version`:** A constant from the module, incremented when the governance schema changes.

4. **Generate `_gov_write_token`:** Compute HMAC or generate unique token. For the existence-constraint-only variant (without trigger verification), this is a UUID. For the APOC-trigger-enhanced variant, this is an HMAC.

5. **Set `_gov_provenance_epoch`:** `new Date().toISOString()` at creation time.

6. **Include all five properties in the MERGE/ON CREATE SET clause** of the atomic write transaction.

### 7.2 Impact on `updateMorpheme()`

Governance properties are **immutable after creation**. The mutation resonator must:

- Reject any update attempt that includes `_gov_*` properties in the `updates` parameter
- Add `_gov_*` properties to the "protected properties" list alongside `id`

This is a straightforward guard addition.

### 7.3 Impact on `stampBloomComplete()`

The stamp protocol calls `updateMorpheme()` internally, which does not create new nodes. No governance property changes needed. The INSTANTIATES backfill logic (Step 5 of stamp) calls `createLine()`, which operates on relationships, not node properties.

### 7.4 Impact on `recordInstantiationObservation()` and Similar Helpers

These functions create observation Seeds via raw `CREATE` in `writeTransaction`. After this change, **they must also provide governance properties** for the observation Seeds they create. This is a significant change — every internal helper that creates any morpheme-labeled node must go through the governance property generation path.

**Design decision:** Either:
- **(a)** Route all observation recording through `instantiateMorpheme()` recursively (creates infinite recursion on observation-of-observation; must be bounded)
- **(b)** Give observation helpers a dedicated "governance bypass" that generates governance properties but doesn't record its own observation (breaks the "all writes go through governance" invariant at the observation layer)
- **(c)** Create observation nodes with a different label that is NOT subject to governance constraints (e.g., `_ObservationRecord` instead of `Seed` with `seedType: 'observation'`)

Option **(c)** is the cleanest: it acknowledges that governance infrastructure nodes (the observation records themselves) cannot be governed by the same mechanism without recursion. This creates a two-tier node model: *governed morphemes* (the five labels with constraints) and *governance infrastructure* (observation records, instantiation records) with a separate, simpler constraint set.

### 7.5 Impact on `ensureViolationGrid()` and Bootstrap Helpers

The `ensureViolationGrid()` function creates a Grid node via raw `MERGE`. After this change, it must provide all governance properties. But this is a bootstrap function — it runs before the governance layer is fully operational.

**Bootstrap paradox:** The governance properties require the governance layer to generate them. The governance layer's infrastructure nodes (Grids, observation Grids) require governance properties to be created. Neither can exist first.

**Resolution:** The migration function (`migrateSchema`) must create bootstrap nodes *before* the governance property constraints are applied. The constraint DDL must run *after* bootstrap. This creates a fixed ordering dependency in the migration:

1. Apply non-governance constraints (existing schema)
2. Create bootstrap infrastructure nodes (Constitutional Bloom, definition Seeds, observation Grids, violation Grid)
3. Apply governance property constraints
4. All subsequent writes go through governed path

Existing nodes created before the governance constraints are applied will need a one-time backfill of governance properties (with a sentinel value like `_gov_instantiation_id: 'bootstrap:pre-governance'`). The uniqueness constraint means each pre-existing node needs a unique sentinel.

## 8. Honest Assessment: Does Option A Satisfy the Requirements?

| Requirement | Option A Alone | Option A + APOC Trigger |
|---|---|---|
| **(a) Structural enforcement, not detection** | **Partial.** Prevents creation without governance properties. Does NOT prevent creation with fake governance properties. | **Yes.** Trigger verifies referential integrity pre-commit. |
| **(b) No monitoring overlay** | **Yes.** Constraints are evaluated at write time, inline with the transaction. | **Yes.** Trigger is pre-commit, synchronous, transaction-blocking. Not a periodic scan. |
| **(c) Works when application layer is compromised** | **Partial.** Constraints enforce property presence regardless of application state. But a compromised app can generate valid-looking fake values. | **Partial.** Trigger runs at DB layer (survives app compromise). But if app is compromised, the HMAC key may be extracted, allowing forgery of write tokens. FK check on InstantiationRecord survives if attacker can't create those records — but if app is compromised, they can. |

### 8.1 The Irreducible Limitation

**Option A, even with all enhancements, cannot fully satisfy requirement (c).** Here's why:

If the application layer is compromised, the attacker has access to:
- The governance property generation code
- Any secrets (HMAC keys, signing keys) stored in the application
- The `instantiateMorpheme()` function itself (they can call it directly with arbitrary parameters)

The attacker doesn't need to bypass the governance layer — they can *use* it with malicious inputs. The governance layer cannot distinguish "legitimate application calling `instantiateMorpheme()`" from "compromised application calling `instantiateMorpheme()` with attacker-chosen parameters."

**This is a fundamental limit of any defence that depends on a shared secret between the application and the database.** True defence against application compromise requires either:
- A secret the application never possesses (hardware security module / external signing service)
- Multi-party authorization (the database requires signatures from multiple independent systems)
- Capability-based access control where the database issues one-time write tokens that are scope-limited (Neo4j does not support this natively)

### 8.2 What Option A Actually Provides

Despite the limitations, Option A provides meaningful defence-in-depth:

1. **Raises the bar significantly.** An attacker must know the governance property schema, provide valid-looking values for all properties, and generate unique values that satisfy uniqueness constraints. Casual injection via copy-pasted Cypher fails.

2. **Creates forensic distinguishability.** Legitimate nodes have governance properties that reference real InstantiationRecord nodes. Injected nodes have governance properties that reference nothing. A single query can partition the graph into governed and ungoverned nodes:
   ```cypher
   MATCH (s:Seed)
   WHERE NOT EXISTS {
     MATCH (ir:InstantiationRecord {id: s._gov_instantiation_id})
   }
   RETURN s.id
   ```
   **This is detection, not enforcement** — but it's detection that can be run after any suspected compromise to identify every injected node. The governance properties make injected nodes *permanently marked* even if they look legitimate by other properties.

3. **Prevents the trivial case.** The most likely attack vector (a quick `CREATE (:Seed {...})` in a Neo4j Browser session) fails immediately. The attacker gets a constraint violation error and must research the governance schema to proceed.

4. **Enables stronger mechanisms.** The governance properties are the *foundation* on which APOC triggers, HMAC verification, or external signing services can build. Without mandatory properties, there's nothing for a trigger to verify.

## 9. Recommendations

### 9.1 Adopt Option A as Necessary Foundation, Not Sufficient Defence

Option A should be implemented as the **structural foundation** of the perimeter defence. It is necessary but not sufficient. It must be combined with at least one of:

- **APOC triggers** (Option A + 5.2.A) for environments where APOC is available
- **Neo4j RBAC** with restricted write roles that only the application service account possesses (reduces Class A attack surface)
- **External signing service** for `_gov_write_token` generation (addresses requirement (c) by removing the secret from the application layer entirely)

### 9.2 Do Not Treat Constraints as Complete Solution

The fake-value attack analysis (§4) demonstrates that existence constraints alone are necessary but insufficient. Any communication about this defence must be honest about its limitations. Claiming that "all morpheme nodes have governance properties, therefore the perimeter is secure" would be false.

### 9.3 Implement the Bootstrap Ordering Carefully

The bootstrap paradox (§7.5) requires careful migration ordering. Governance property constraints must be the **last** schema step, applied only after all infrastructure nodes exist. The `migrateSchema()` function must be restructured into phases.

### 9.4 Separate Governance Infrastructure from Governed Morphemes

Observation Seeds and InstantiationRecord nodes should use labels outside the governed set (§7.4 option (c)). This avoids recursion in the governance protocol and clearly delineates the two tiers of the graph: nodes that are governed and nodes that are governance infrastructure.
